# API 연동 이슈 정리 (5) — UI/UX 점검 4번 항목 수정 중 발견

> `docs/ui-ux-audit.md`의 "4. 장바구니 / 결제 / 주문"을 고치는 과정에서 확인한 것들. 4-1, 4-3~4-6은 프론트
> 로직/렌더링만으로 해결 완료(백엔드 연관 없음). 아래 두 가지만 백엔드 계약과 관련이 있어 별도로 남긴다.

## 1. 결제수단이 하나뿐인 건 프론트 하드코딩이지, 백엔드가 여러 개를 지원 안 하는 게 확인된 건 아님

- **한 것**: `CheckoutPage.tsx`에서 결제수단을 두 번(회색 카드 + 흰 배경 가짜 라디오) 중복 표시하던 걸 하나로
  합쳤다 (4-2).
- **남는 것**: `PAYMENT_METHOD = '앱팡 머니'`는 프론트 상수이고, `OrderCreateRequest.paymentMethod`는
  `string` 타입이라 백엔드가 다른 값을 받아줄 수도, 안 받아줄 수도 있다. 결제수단 목록을 조회하는 API가
  없어서 "앱팡 머니만 지원하는지, 다른 결제수단도 있는데 프론트가 안 붙였는지"는 확인 불가능하다.
- **필요한 것**: 결제수단이 여러 개로 늘어날 계획이 있다면 `GET /api/payment-methods` 같은 목록 API가 있어야
  진짜 선택 UI(라디오)를 다시 넣을 근거가 생긴다. 지금처럼 하나뿐이면 현재 상태(텍스트 표시)가 맞다.

## 2. 주문상세 "리뷰 쓰기" 버튼 노출 조건이 검증 안 된 status 문자열에 의존함

- **한 것**: `OrderDetailPage.tsx`에 배송완료 아이템에 한해 "리뷰 쓰기" 버튼을 추가해 상품상세로 이동하도록
  연결했다 (4-7). 조건은 `item.status === '배송완료'`.
- **문제**: `docs/api-integration-issues.md` 3번 항목에 이미 적혀 있듯, 주문 `status` 값이 실제로
  `배송완료`/`배송중`/`주문접수` 한글 문자열과 정확히 일치하는지 검증되지 않았다. `OrderListPage`/`MyPage`는
  값이 안 맞을 때 `주문접수`로 폴백하는 `toStatus()` 방어 코드가 있지만, `OrderDetailPage.tsx`의
  `order.items` 렌더링은 `item.status`를 폴백 없이 그대로 쓰고 있어서(이번에 추가한 조건도 동일), 백엔드 값이
  스펙과 다르면 배송이 끝난 상품이어도 "리뷰 쓰기" 버튼이 영영 안 뜰 수 있다.
- **추가로 발견한 것**: 상품상세 페이지(`ProductDetailPage`)는 "리뷰 작성 가능 여부"를 백엔드가 내려주는
  `ProductDetailResponse.canWriteReview` 값으로 판단하는데, 이번에 주문상세에 추가한 버튼은 완전히 다른
  신호(주문 아이템의 `status` 문자열)로 판단한다. 두 화면이 같은 질문("이 상품에 리뷰를 쓸 수 있는가")에
  서로 다른 기준을 쓰고 있어서, 이론상 한쪽에서는 버튼이 보이는데 실제 상품상세에 들어가면
  `canWriteReview: false`라 리뷰 작성이 막혀 있는 경우가 생길 수 있다.
- **필요한 것**: 실 데이터로 `status` 값을 확인해 `toStatus()`류 폴백을 `OrderDetailPage`에도 동일하게
  적용하거나, 더 근본적으로는 백엔드가 주문 아이템 응답에 `canWriteReview` 같은 명시적 플래그를 내려줘서
  두 화면이 같은 기준으로 판단하게 하는 걸 권장한다.
