# API 연동 이슈 정리 (7) — UI/UX 점검 6번 항목 수정 중 발견

> `docs/ui-ux-audit.md`의 "6. 로그인 / 회원가입 / 설정"을 고치는 과정에서 확인한 것. 6-1, 6-2, 6-4 모두
> 프론트 상태/렌더링만으로 해결 완료(백엔드 연관 없음). 6-3은 이미 1번 항목 작업 때 "준비 중" 토스트로
> 처리돼 있어 이번엔 손대지 않았다. 백엔드 이슈는 없고, 작업 중 눈에 띈 것 하나만 남긴다.

## 1. (백엔드 이슈 아님) 토스트 통일 작업 내용

`src/components/Toast.tsx`를 새로 만들어 로그인/회원가입/설정/마이페이지/주문내역/상품상세/배송지 목록에
흩어져 있던 토스트 마크업을 전부 이걸로 교체했다. 자동 소멸 2초 + 닫기 버튼을 모든 화면에서 동일하게
동작하도록 맞췄다(기존엔 로그인 실패 토스트와 설정 화면 토스트만 자동으로 안 사라졌음). API 연동과는
무관한 순수 프론트 리팩터링.

## 2. 설정 화면 이메일 변경에는 회원가입 화면과 달리 중복 확인이 없음

- **위치**: `SettingsPage.tsx`의 `handleUpdateEmail` vs `RegisterPage.tsx`의 `handleEmailBlur`
- **현황**: 회원가입 화면은 이메일 입력 후 blur 시 `checkEmail(email)`을 호출해 "이미 사용 중인
  이메일입니다"를 미리 알려주는데, 설정 화면의 이메일 변경은 그런 사전 확인 없이 바로
  `updateMe({ email })`을 호출한다. 실패하면 그냥 "이메일 변경에 실패했습니다"라는 뭉뚱그린 토스트만
  뜬다.
- **왜 지금 안 고쳤나**: `docs/ui-ux-audit.md` 6번 항목엔 없던 내용이라 이번 스코프 밖이라고 판단했다.
  다만 고치려면 백엔드 쪽엔 변경이 필요 없고(`checkEmail` API가 이미 있음), `SettingsPage`에도 회원가입과
  같은 blur 검증만 추가하면 되는 순수 프론트 작업이라는 점만 기록해 둔다.
