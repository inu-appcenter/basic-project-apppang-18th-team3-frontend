# API 연동 이슈 정리 (4) — UI/UX 점검 3번 항목 수정 중 발견

> `docs/ui-ux-audit.md`의 "3. 상품 목록 / 상품 상세"를 고치는 과정에서 확인한 백엔드 계약 제약. 3-1~3-4 모두
> 프론트 코드/데이터만으로 해결은 완료했지만, 아래 두 가지는 백엔드가 아직 못 받쳐주는 부분이라 한계가 남는다.

## 1. 수량 옵션은 실제 quantity에 반영했지만, 색상 옵션은 여전히 화면 표시용일 뿐

- **한 것**: `QUANTITY_OPTIONS`에 `quantity: 20/40/60` 숫자를 추가해서, 장바구니 담기/바로구매 시 사용자가 고른
  옵션 그대로 `addToCart`/`checkoutStore`에 전달되도록 연결했다 (`ProductDetailPage.tsx`).
- **한계**: `COLOR_OPTIONS`(화이트+그레이/화이트/그레이)는 여전히 선택 상태(`selectedColor`)만 로컬로 갖고 있고
  어디로도 전송하지 않는다. `ProductDetailResponse`/`CartAddRequest`/`OrderItemRequest` 어디에도 색상·옵션 값을
  실을 필드가 없어서(문자열 `optionInfo` 하나뿐), 프론트에서 만들어낼 수 있는 필드가 없다.
- **필요한 것**: 실제 옵션별 재고/가격이 다르다면 백엔드에 옵션(변형) 테이블과, 장바구니/주문 요청에 옵션 ID를
  실을 필드가 추가돼야 한다. 그전까지는 색상 선택 UI가 "고를 수는 있지만 결제엔 반영 안 되는" 상태로 남는다 —
  `ui-ux-audit.md` 3-1에 이미 적힌 대로 데모 데이터라는 것만 재확인.

## 2. "로켓배송만 보기" 토글은 서버 필터가 아니라 클라이언트 필터

- **한 것**: `ProductListPage.tsx`에 토글 버튼을 추가하고, 이미 불러온 `products` 배열을 `isRocket` 기준으로
  화면에서만 걸러서 보여주도록 구현했다(서버 요청 파라미터는 그대로).
- **왜 서버 필터로 안 했나**: `docs/api-integration-issues.md` 3번 항목에 이미 적혀 있듯 `GET /api/products`의
  쿼리 파라미터가 Swagger에 문서화돼 있지 않아, `rocketOnly` 같은 파라미터가 실제로 존재하는지 확인할 방법이
  없었다. 임의로 파라미터를 추가해 호출하면 백엔드가 무시하거나 400을 낼 수 있어 클라이언트 필터로 우회했다.
- **부작용**: 무한스크롤 센티널은 필터링 이전의 원본 목록 기준으로 다음 페이지를 계속 불러오기 때문에, 로켓배송
  상품이 드문 카테고리에서는 "더 볼 상품 없음" 판정 전까지 여러 페이지를 백그라운드로 계속 불러오게 된다.
  데이터 양이 많아지면 트래픽 낭비로 이어질 수 있어, 상품 수가 늘어나면 서버 사이드 필터 파라미터로 옮기는 걸
  권장한다.
